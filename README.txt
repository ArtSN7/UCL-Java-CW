Initially, the user is redirected to the Home page, where they can view the main dashboard. Since the CSV file contains many fields, I chose to display only the most relevant columns in the table so the data remains clear and easy to read.

From the dashboard, the user can click on the ID field in any row to view the details of that specific patient. This opens a new page showing the full information for the selected patient. On that page, the user can update any of the patient’s details and click Save to write the changes back to the CSV file. The user can also delete the patient if needed. Both updating and deleting trigger a rewrite of the CSV file.

Next to the table, there are also two additional actions: Add Patient and Export to JSON. The Export to JSON feature exports all patient records from the CSV file into a JSON file. The Add Patient button opens a separate page where the user can add a new patient by entering the required details. The patient ID is generated automatically. Once the new patient is added, the system rewrites the CSV file and updates the UI accordingly.

Search Feature

- The search feature allows users to enter one or more keywords and returns matching patient records in a table view.

The search query is first normalized by trimming whitespace, converting the text to lowercase, and splitting it into keywords based on spaces.

For each CSV row, the system combines all column values into one lowercase searchable string. It then applies AND-based keyword matching, meaning a row is returned only if every keyword appears somewhere in that row’s text.

The matching is:
	•	case-insensitive
	•	based on substring comparison
	•	performed using an in-memory linear scan over the loaded DataFrame

For display purposes, only selected columns are returned in the results table: ID, FIRST, LAST, CITY, and STATE. Not all raw CSV fields are shown.

Statistics Page

What it does

The statistics page (/statistics) provides:
- Summary metrics:
	•	oldest living person
	•	youngest living person
	•	oldest overall person
	•	youngest overall person
	•	counts of alive, dead, and total records
	•	City filtering:
	•	displays names of people in a selected city
	•	supports an optional status filter: all, alive, or dead
	•	Demographic charts:
	•	age distribution (bar chart)
	•	ethnicity distribution (pie chart)
	•	race distribution (pie chart)

Core logic used

The statistics summary is generated by iterating through all rows once. The system determines whether a person is alive or deceased based on the DEATHDATE field, where a blank value indicates the person is alive. It builds PersonMetric objects from the row data and computes age as follows:
	•	For deceased individuals: Period.between(birthDate, deathDate)
	•	For living individuals: Period.between(birthDate, LocalDate.now())

Oldest and youngest values are tracked using helper comparison methods.

Note:*
	•	rows with missing or invalid BIRTHDATE values are skipped for age-based metrics
	•	rows with invalid DEATHDATE values, or where death occurs before birth, are also skipped

For city filtering:
	•	the city value must not be blank
	•	the system applies a life-status filter enum: ALL, ALIVE, or DEAD

For the charts:
	•	age distribution is built using a sorted map by numeric age
	•	ethnicity and race are grouped case-insensitively
	•	blank ethnicity or race values are grouped as Unknown
